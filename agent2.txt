  1. 핵심 패턴: OpenAI Assistants API + Function Calling

  OpenAI Assistants API를 사용한 구조입니다. 단순 Chat Completion이 아닌 Thread 기반 대화 관리 방식입니다.

  사용자 메시지
      ↓
  Thread (대화 맥락 저장)
      ↓
  Run (실행)
      ↓
  [requires_action] → Function Call 실행 → Tool 결과 반환
      ↓
  [completed] → 최종 응답 반환

  4. 3개 Function Tool 구조 (공통 패턴)

  각 Tool은 동일한 패턴으로 구현:

  # 1) OpenAI Function 스키마 정의 (JSON Schema)
  SEARCH_PETS_FUNCTION = {
      "type": "function",
      "function": {
          "name": "search_pets",
          "description": "...",
          "parameters": { ... }
      }
  }

  # 2) Tool 클래스 (실제 비즈니스 로직)
  class PetSearchTool:
      def search_pets(self, user_preferences: str) -> Dict: ...

  # 3) 실행 함수 (Assistant가 호출하는 진입점)
  def execute_pet_search(user_preferences: str) -> str:
      result = pet_search_tool.search_pets(user_preferences)
      return json.dumps(result, ensure_ascii=False)  # 반드시 JSON 문자열 반환

  Tool별 알고리즘

  pet_search_tool.py - 강아지 검색
  1. 백엔드 API 호출 (/api/pets?adopted=false)
  2. 2단계 필터링: 필수조건(나이 이상, 지역) → 선택조건(점수화)
  3. 점수 기준: 품종(+3), 크기(+2), 성격(+4/-2), 지역(+2), 성별(+1) 등
  4. 상위 3마리 반환

  insurance_recommendation_tool.py - 보험 추천
  1. 백엔드 API 호출 (/api/insurance)
  2. 품종별 위험질병 DB(하드코딩) 기반 매칭
  3. 점수 기준: 품종 위험질병 보장(+3), 나이대 적합성(+2~3), 크기(+1.5), 수술/입원비(+1), 대형 보험사(+0.5)
  4. 상위 3개 반환

  product_recommendation_tool.py - 상품 추천
  1. NaverProduct DB 검색 (키워드 기반 15개 제한)
  2. 부족하면 일반 상품으로 Fallback
  3. 다양성 알고리즘: 상위30% → 중위40% → 나머지에서 카테고리별 다양하게 선택
  4. 6개 상품 반환

  면접 포인트로 강조할 부분

  1. OpenAI Assistants API vs Chat Completion: Thread로 대화 맥락을 서버에서 관리 → 클라이언트가 히스토리를 직접 관리할 필요 없음
  2. Polling 방식: requires_action 상태 감지로 Function Call 처리 → 비동기적으로 여러 Function을 순차 실행 가능
  3. 단계별 상태 관리: 세션에 AgentStage를 두어 입양 상담 플로우를 추적
  4. Rule-based 필터링 + 점수화: LLM에 의존하지 않고 자체 로직으로 추천 품질 제어
  5. 한계점: 세션을 인메모리(Dict)로만 관리 → 서버 재시작 시 세션 소멸, 멀티 인스턴스 환경에서 공유 불가

  1단계: 호출 가능한 함수를 GPT에게 알려주는 부분

  adoption_assistant.py:48-95 — create_assistant()

  호출해야 하는지를 가이드합니다.

  실제 호출 결정은 OpenAI 모델 내부에서 일어납니다. 코드에는 없고, GPT가 대화 맥락을 보고 스스로 판단합니다.

  2단계: GPT가 호출을 결정했다는 신호를 포착하는 부분

  adoption_assistant.py:168-170 — _wait_for_completion()

  elif run.status == "requires_action":

  - 어떤 함수: function_name → "search_pets"
  - 어떤 인자: function_args → {"user_preferences": "서울에 온순한 소형견"}

  create_assistant(tools=[...])

  ---
  submit_tool_outputs 이후 흐름

  _handle_function_calls()가 submit_tool_outputs를 호출하고 함수가 종료됩니다. 그러면 호출한 곳인 _wait_for_completion()으로 돌아옵니다.     

  adoption_assistant.py:147-179

  for _ in range(max_iterations):          # ← 루프가 계속 돌고 있음
      run = self.client.beta.threads.runs.retrieve(...)

      if run.status == "completed":
          # ...
      elif run.status == "requires_action":
          self._handle_function_calls(...)  # ← 여기서 submit_tool_outputs 호출
          # 함수 종료 후 → 아무것도 없음

      time.sleep(1)                         # ← 1초 대기 후 루프 다시 실행

  _handle_function_calls()가 끝나면 time.sleep(1) 실행 후 루프가 다시 돌아서 runs.retrieve()를 한 번 더 호출합니다.

  ---
  그 사이에 무슨 일이 일어나는가

  submit_tool_outputs() 호출
      ↓
  [OpenAI 서버 내부]
      GPT가 tool_outputs 결과를 받음
      → 결과를 바탕으로 자연어 응답 생성
      → Run 상태를 requires_action → completed 로 변경
      ↓
  time.sleep(1) → 루프 재실행
      ↓
  runs.retrieve() → status: "completed"
      ↓
  messages.list() → GPT가 생성한 자연어 응답 꺼냄

  즉, submit_tool_outputs()를 호출하는 순간 OpenAI 서버에서 자동으로 Run이 재개되고, GPT가 tool 결과를 입력으로 받아 자연어 응답을
  생성합니다. 코드가 능동적으로 "GPT야 이제 설명해줘"라고 호출하는 게 아니라, 다음 폴링 타이밍에 이미 completed가 되어있는 것을 확인하는     
  구조입니다.

  ---
  ReAct와의 차이

  ReAct는 LLM이 텍스트로 추론 과정을 명시적으로 출력하는 패턴입니다:
  Thought: 사용자가 소형견을 원하니 search_pets를 호출해야겠다
  Action: search_pets
  Action Input: {"user_preferences": "소형견"}
  Observation: [결과]
  Thought: 이제 보험을 추천해야겠다
  ...

  Tool Calling은 추론 과정 없이 구조화된 JSON으로 바로 반환합니다:
  {"name": "search_pets", "arguments": {"user_preferences": "소형견"}}

  ---
  정리

  현재 코드는:
  - OpenAI Assistants API → 대화 상태(Thread)를 서버에서 관리
  - Tool Calling (Function Calling) → GPT가 JSON으로 함수 호출을 결정

  LangChain 없이 이 두 가지를 직접 구현한 방식입니다. 제가 ReAct라고 한 것은 잘못된 표현이었습니다.

────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

❯ 핵심 패턴: OpenAI Assistants API + Function Calling 이라고 맨 처음에 설명해줬잖아                                                          
  이게 tool calling 방식이라고 생각하면 되는거야?                                                                                            

● 네, 맞습니다.

  Function Calling = Tool Calling 입니다. 같은 개념을 부르는 두 가지 이름입니다.

  - OpenAI가 초기에 Function Calling 이라고 명명했다가
  - 이후 공식 문서에서 Tool Calling 으로 용어를 통일했습니다

  그래서 코드에서 tools=[...]로 등록하고, GPT가 JSON으로 호출 결정을 반환하는 이 방식을 Tool Calling (= Function Calling) 이라고 부르면      
  됩니다.

  면접에서는 둘 다 같은 의미로 통하니 어느 쪽으로 말해도 무방합니다.
